Introduction to Kubernetes

‹ Back


In order to facilitate a platform for deploying applications that is highly available, the underlying architecture needs to be designed in a way that provides 100% uptime. Kubernetes is designed as a highly available cluster of computers that are connected to work as a single unit. This abstraction allows us to deploy applications without thinking on what specific machines we need to do it. To make use of this new type of deployments, applications need to be packaged in a standard way: they need to be containerized. This is different compared to how applications were deployed in the past, when they were directly installed on specific machines as binaries deeply integrated into the host.

Kubernetes is an open-source platform for scheduling and running applications packaged in containers across computer clusters.

A Kubernetes cluster is formed out of 2 types of resources:
masters - are managing the cluster
nodes - are where we run applications

Main concepts

  • cluster
  • master
  • node
  • minikube

Kubernetes is an open-source platform for scheduling and running applications packaged in containers across computer clusters.


Cluster Diagram


Masters are responsible for managing the activity of the entire cluster. The master nodes will coordinate all the activity happening in your cluster like scheduling applications, maintaining their desired state, scaling applications and rolling new updates. In order to assure that your cluster is highly available, you need to run multiple master instances (minimum 3).

A node is a worker machine in Kubernetes, all your applications will run on nodes. Every node from the cluster is managed by the masters. On a typical node you will have tools for handling container operations (like Docker, rkt) and Kubelet, a tool for managing the node activity. To have a highly available cluster for your applications, you will need to have at least 3 nodes available.

Deploying applications on Kubernetes means that you’ll ask a master to schedule your containers to run on some node agents according to your desired specifications. Communications between the 2 type of nodes is done via an API. The same API is exposed towards the users in order to facilitate interaction with the cluster.

The name Kubernetes originates from Greek, meaning “helmsman” or “pilot”, and is the root of the words “governor” and “cybernetic”. K8s is an abbreviation obtained by replacing the 8 letters “ubernete” with 8.

Kubernetes can be installed both on physical or virtual machines, and can be deployed on your laptop, on your datacenter or in the cloud. The easiest (and recommended) way to run kubernetes on a development box is with minikube which will create on your local machine a VM with a simple cluster that has one resource node. Minikube is available for Linux and Mac OS systems, with Windows support coming in the next major release. The minikube CLI exposes the basic operations we can expect from a bootstrapping tool like: start, stop, status, delete. To keep things simple during this bootcamp, we’ll use an online terminal where we have already installed minikube.

Now that you know what Kubernetes is, let’s go to the online tutorial and start our first cluster!

Masters are managing the cluster and the nodes are used to host the running applications.
